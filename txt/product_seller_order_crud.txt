# Product Seller Mobile Order APIs

## 1. Get Product Order List
Retrieves a paginated and filterable list of orders for the authenticated product seller.

- **Endpoint**: `GET /mobileapi/product-seller/orders`
- **Authentication**: Required (Product Seller session)

### Query Parameters:
- `page` (number, default: 1): The page number.
- `perPage` (number, default: 10): Items per page.
- `orderNumber` (string, optional): Filter by order number (strips leading # automatically).
- `customerName` (string, optional): Filter by customer name.
- `email` (string, optional): Filter by customer email.
- `productName` (string, optional): Filter by product name.
- `startDate` (string, optional): ISO date string (YYYY-MM-DD).
- `endDate` (string, optional): ISO date string (YYYY-MM-DD).
- `status` (string, optional): Filter by order item status (PENDING, SHIPPED, DELIVERED, etc.).

### Response Structure:
```json
{
  "orders": [
    {
      "id": "order_id",
      "orderNumber": "meeem00000008",
      "status": "PENDING",
      "totalAmount": 1265.00,
      "createdAt": "2026-04-22T15:00:00Z",
      "customer": {
        "name": "Arijit",
        "email": "arijit@example.com"
      },
      "items": [
        {
          "id": "item_id",
          "productNameSnapshot": "Fashion - Men's Clothing Example",
          "quantity": 1,
          "price": 100.00,
          "imageUrl": "https://...",
          "status": "PENDING"
        }
      ]
    }
  ],
  "totalCount": 1,
  "totalPages": 1,
  "page": 1,
  "perPage": 10
}
```

## 3. Update Product Item Status
Updates the fulfillment status of a product item (Dispatcher Hub).

- **Endpoint**: `PATCH /mobileapi/product-seller/orders/[id]`
- **Authentication**: Required (Product Seller session)

### Request Payload:
```json
{
  "status": "SHIPPED", // PENDING, CONFIRMED, PROCESSING, SHIPPED, OUT_FOR_DELIVERY, DELIVERED, CANCELLED
  "itemId": "item_id_1",
  "itemIds": ["item_id_1", "item_id_2"],
  "location": "Warehouse A",
  "note": "Packed",
  "otp": "123456", // Required for DELIVERED if previously OUT_FOR_DELIVERY
  "deliveryProofImage": "https://..." // Required for DELIVERED
}
```

### Business Logic & Guards:
1.  **Backward Movement Protection**: Blocks moving to a past status.
2.  **OTP Sending**: Moving to `OUT_FOR_DELIVERY` automatically sends OTP to customer.
3.  **Delivery Verification**: `DELIVERED` requires valid `otp` and `deliveryProofImage`.
4.  **Cancellations**: Only allowed from `PENDING` and if no other item in order is `DELIVERED`.

# Return & Exchange Mobile APIs

### 💡 Frontend Integration Note: Checking Return Eligibility
Before calling any of the return/exchange APIs below, the mobile app must determine if the "Return/Exchange" button should be visible. 
You do **NOT** need a new API for this. This data is already provided in the existing **Customer Order Details API** (`GET /mobileapi/customer/orders/[id]`).

Inside the order details response, every item contains the following fields:
```json
{
  "returnAvailable": true,
  "replacementAllowed": true,
  "returnPolicyType": "RETURNABLE",
  "returnPolicyDays": 7,
  "statusHistory": [ ... ]
}
```
**Logic to show the Return button:**
Show the button ONLY IF:
1. `returnAvailable` is `true`.
2. The item's current status is `DELIVERED`.
3. The current date is before the deadline (Calculate deadline by finding the `createdAt` timestamp of the `DELIVERED` status in `statusHistory`, and adding `returnPolicyDays`).
If any of these are false, hide the button entirely.

## 5. Submit Return/Exchange Request (Customer)
Allows a customer to request a refund or an exchange for a delivered item within the return window.

- **Endpoint**: `POST /mobileapi/customer/orders/products/details/[id]/items/[orderItemId]/return-request`
- **Authentication**: Required (Customer Bearer Token)

### Request Payload (Supports multipart/form-data & JSON):

**Option A: Direct Image Upload (Recommended for Mobile)**
Send as `multipart/form-data`:
- `reason` (text): "Item is defective"
- `returnImages` (files): Multiple image files appended to the same key.
- `resolutionType` (text): "EXCHANGE" or "REFUND"
- `replacementVariantId` (text): "var_2" (Required if EXCHANGE)

*Example Mobile Request (React Native / Fetch):*
```javascript
const formData = new FormData();
formData.append("reason", "Item arrived broken");
formData.append("resolutionType", "REFUND");
// Append multiple files
formData.append("returnImages", { uri: "file://...", name: "photo1.jpg", type: "image/jpeg" });
formData.append("returnImages", { uri: "file://...", name: "photo2.jpg", type: "image/jpeg" });

fetch("https://api.../return-request", {
  method: "POST",
  headers: { "Authorization": "Bearer YOUR_TOKEN" },
  body: formData
});
```

**Option B: Existing URLs (JSON)**
If the images are already uploaded, send as `application/json`:
```json
{
  "reason": "Item is defective",
  "returnImages": ["https://image1.jpg", "https://image2.jpg"],
  "resolutionType": "EXCHANGE",
  "replacementVariantId": "var_2"
}
```

### Business Logic & Validations:
When this API is called, the backend automatically checks:
1. **Delivery Status**: Item must be `DELIVERED`.
2. **Returnable Status**: The item's variant must be `RETURNABLE`.
3. **Return Window**: The current date must be within `deliveredAt + returnDays`.
4. **Stock Check (Exchange)**: If requesting an exchange, the selected `replacementVariantId` must have sufficient stock.
If any check fails, it returns a `400 Bad Request` with an appropriate error message.

### Response Payload:
```json
{
  "success": true,
  "message": "Return request created",
  "data": {
    "id": "req_1",
    "status": "REQUESTED",
    "pickupStatus": "NOT_REQUESTED",
    "refundStatus": "NOT_REQUESTED",
    "resolutionType": "EXCHANGE"
  }
}
```

## 6. Manage Return/Exchange Request (Product Seller)
Allows the seller to accept/reject a return request and manage its lifecycle (pickup, refund, top-up). 

- **Endpoint**: `PATCH /mobileapi/product-seller/orders/[id]/items/[orderItemId]/return-request`
- **Authentication**: Required (Product Seller Bearer Token)

### Request Payload:
```json
{
  "action": "ACCEPT" // Allowed: "ACCEPT", "REJECT", "PICKUP_COMPLETED", "REFUND_COMPLETED", "EXCHANGE_TOP_UP_RECEIVED"
}
```

### Response Payload (Example: ACCEPT on an Exchange):
```json
{
  "success": true,
  "message": "Exchange accepted",
  "data": {
    "id": "req_1",
    "status": "ACCEPTED",
    "pickupStatus": "PENDING",
    "refundStatus": "NOT_REQUESTED",
    "resolutionType": "EXCHANGE",
    "replacementOrderItemId": "new_item_id_2",
    "exchangeTopUpAmount": 23.00,
    "exchangeTopUpStatus": "PENDING",
    "exchangeRefundDifferenceAmount": 0,
    "exchangeRefundDifferenceStatus": "NOT_REQUESTED"
  }
}
```

### Business Logic Notes on Actions:
- **`ACCEPT` (Exchange)**: Automatically generates the new replacement order item line (`replacementOrderItemId` will be populated).
- **`ACCEPT` (Refund)**: Updates status to `ACCEPTED`.
- **`PICKUP_COMPLETED` (Refund)**: Automatically triggers full financial credit to the customer's wallet and debits the seller balance.
- **`PICKUP_COMPLETED` (Exchange)**: Just marks the pickup as complete (the exchange completion logic waits until the replacement item is delivered).
- **`EXCHANGE_TOP_UP_RECEIVED`**: Marks top-up as paid and credits seller wallet.
- **`REFUND_COMPLETED`**: Re-stocks the inventory and leaves an audit trail in `statusHistory`.

### Lifecycle Example (Customer Requests Refund):
1. Customer `POST`s return-request with `resolutionType: "REFUND"`.
2. Seller `PATCH`es action `ACCEPT`. `pickupStatus` and `refundStatus` become `PENDING`.
3. Seller `PATCH`es action `PICKUP_COMPLETED`. `pickupStatus` becomes `COMPLETED` and `refundStatus` becomes `COMPLETED` automatically (funds transfer).
4. Seller `PATCH`es action `REFUND_COMPLETED`. Inventory is restocked, history updated to `REFUNDED`.
