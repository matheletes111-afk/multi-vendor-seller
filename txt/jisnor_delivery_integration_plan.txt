================================================================================
   INTEGRATION PLAN: MULTI-PARTNER & SELF-DELIVERY LOGISTICS
   Merging with Product & Food Delivery Systems (Multi-Vendor Marketplace)
================================================================================

Currently, all delivery logistics, costs, and statuses are handled manually by the individual sellers (Product Sellers & Restaurant/Food Sellers) under a "Self-Delivery" model. 

This document outlines a modular, extensible integration plan to support:
1. Seller Self-Delivery (Existing manual model, customizable by seller)
2. Jisnor4U Delivery (Tech-enabled, standard delivery partner)
3. Extensible Multi-Partner Framework (To easily onboard other carriers like FedEx, DHL, or local courier services in the future)

---

1. ARCHITECTURAL OVERVIEW: MODULAR ADAPTER PATTERN
--------------------------------------------------------------------------------
To support multiple delivery providers (including self-delivery) without cluttering our checkout or booking business logic, we will implement a Strategy/Adapter pattern.

A unified Delivery Service Interface will be defined in our backend (`src/lib/delivery/types.ts`):

```typescript
export interface DeliveryProviderAdapter {
  providerId: string; // e.g., 'SELF', 'JISNOR4U', 'PARTNER_X'
  
  // Calculates delivery fee based on pickup location, customer location, and items
  calculateFee(params: {
    origin: { lat: number; lng: number; address: string };
    destination: { lat: number; lng: number; address: string };
    items: Array<{ weight?: number; dimensions?: any; price: number }>;
  }): Promise<{ fee: number; estimatedDeliveryTime: string; isServiceable: boolean }>;

  // Triggers dispatch request to the partner
  bookDelivery(order: any): Promise<{
    trackingId: string;
    status: string;
    rawResponse: any;
  }>;

  // Cancels active dispatch request
  cancelDelivery(trackingId: string): Promise<{ success: boolean }>;

  // Optional: Gets real-time tracking coordinates / status
  trackDelivery(trackingId: string): Promise<{
    status: string;
    riderInfo?: { name: string; phone: string; lat?: number; lng?: number };
  }>;
}
```

We will implement:
- `SelfDeliveryAdapter`: Returns configured seller flat fees or per-km rates. Booking is a no-op (status marked as PENDING for the seller to fulfill).
- `Jisnor4uAdapter`: Queries Jisnor4U's APIs for real-time rates and handles API bookings.
- `FuturePartnerAdapter`: Plugs into this interface to call other third-party shipping APIs.


2. STEP-BY-STEP IMPLEMENTATION PLAN
--------------------------------------------------------------------------------

STEP 1: DATABASE & SCHEMA UPDATES
We need to update our database models (Prisma) to store delivery providers, delivery fees, and tracking info.
- Order / FoodOrder Table Updates:
  * deliveryProvider: String @default("SELF") - Stores 'SELF', 'JISNOR4U', etc.
  * deliveryFee: Decimal/Float - The computed delivery charge.
  * deliveryTrackingId: String (optional) - The unique tracking identifier from the carrier.
  * deliveryStatus: Enum ('PENDING', 'ASSIGNED', 'PICKED_UP', 'SHIPPED', 'DELIVERED', 'CANCELLED').
  * deliveryRiderInfo: JSON (optional) - Stores rider name, phone, and current coordinates.
- Seller / Restaurant Profile Updates:
  * longitude & latitude (or exact address coordinates) - Necessary for distance calculation.
  * supportedDeliveryProviders: String[] @default(["SELF"]) - Array of providers enabled by this seller (e.g., ['SELF', 'JISNOR4U']).
  * selfDeliveryFlatRate: Float @default(0) - Flat rate the seller charges if 'SELF' delivery is selected.
  * selfDeliveryPerKmRate: Float @default(0) - Per-kilometer rate if the seller uses distance-based self-delivery.

STEP 2: MODULAR DELIVERY UTILITY (THE DISPATCHER)
- Create a registry utility `src/lib/delivery/dispatcher.ts`:
  * Dynamically resolves the active adapter based on the chosen `deliveryProvider`.
  * Exposes simple wrapper functions: `getDeliveryOptions(sellerId, origin, destination, items)`, `bookDelivery(orderId)`, and `cancelDelivery(orderId)`.

STEP 3: RETAIL PRODUCT CHECKOUT INTEGRATION
- Modify `src/app/checkout/checkout-client.tsx` and `/api/customer/checkout/place-order/route.ts`:
  * Fetch available delivery options for the items in the cart:
    1. Query `/api/delivery/options` passing checkout addresses.
    2. The API looks up the seller's `supportedDeliveryProviders`.
    3. For each supported provider, it calls the corresponding adapter's `calculateFee`.
  * Display a radio selection list to the user:
    - [ ] Seller Self-Delivery (Fee: $X.XX)
    - [ ] Jisnor4U Express Delivery (Fee: $Y.YY)
    - [ ] [Other Partner] Express (Fee: $Z.ZZ)
  * Add the selected provider's `deliveryFee` to the checkout Grand Total.

STEP 4: FOOD/RESTAURANT CHECKOUT INTEGRATION
- Modify `src/app/foods/checkout/page.tsx` and `/api/customer/foods/orders/route.ts`:
  * Fetch active delivery options using the restaurant's location coordinates and the customer's delivery address coordinates.
  * Display available options (e.g. Restaurant's own self-delivery vs. Jisnor4U instant rider).
  * Dynamically calculate and update the total amount.

STEP 5: AUTOMATING DISPATCH (BACKEND TRIGGERS)
- Upon successful payment / Cash on Delivery (COD) confirmation:
  * Retrieve the selected `deliveryProvider`.
  * Invoke `DeliveryDispatcher.bookDelivery(order)`.
  * If provider is `SELF`:
    * Mark order `deliveryStatus` as `PENDING` and notify the seller. No external API call is made.
  * If provider is a partner (e.g., `JISNOR4U`):
    * Make the API dispatch call via the adapter.
    * Save tracking ID and initial delivery status to the database.

STEP 6: REAL-TIME TRACKING & WEBHOOK HANDLERS
- Create modular webhook endpoints:
  * `/api/webhooks/delivery/jisnor4u/route.ts` - Handles Jisnor4U updates.
  * `/api/webhooks/delivery/[provider]/route.ts` - Catch-all or individual routes for other partners.
- In each webhook:
  * Parse status updates and map them to our internal `deliveryStatus` (e.g. `ASSIGNED`, `PICKED_UP`, `DELIVERED`).
  * If status transitions to `DELIVERED`:
    * Trigger OrderItem completion and seller wallet payouts.
- Customer Portal:
  * Update tracking UI (`/customer/orders` and `/foods/orders`):
    * For `SELF` delivery, show simple text-based status tracking (e.g., "Seller is preparing/delivering your order").
    * For `JISNOR4U` or third-party partners, render a live tracking card with rider info and a map if coordinates are available.

STEP 7: SELLER PANEL UPDATES
- For Seller Self-Delivery (`SELF`):
  * The seller has full manual control in their dashboard.
  * Buttons are provided to transition order states: "Mark Packaged" → "Mark Dispatched" → "Mark Delivered" (with optional delivery OTP verification).
- For Partner Deliveries:
  * The seller only needs to click "Ready for Pickup".
  * This action triggers the third-party driver dispatch. All subsequent status transitions (SHIPPED, DELIVERED) are handled automatically via webhooks.

3. REVENUE & PAYMENT SPLITTING MODEL
--------------------------------------------------------------------------------
- **For Partner Delivery (e.g., Jisnor4U):**
  * The customer pays the delivery fee to the platform.
  * The platform retains the delivery fee to settle bills with the partner.
  * The seller ledger (`src/lib/restaurant-ledger.ts`) only credits the seller for product/food value (minus commission), excluding the delivery fee.
- **For Seller Self-Delivery:**
  * The customer pays the delivery fee to the platform.
  * The platform credits the delivery fee directly to the seller's net balance (possibly minus a minor platform handling fee), as the seller is handling the transit costs.


4. CRITICAL BUSINESS & TECHNICAL CONSIDERATIONS
--------------------------------------------------------------------------------

A. Multi-Vendor Cart Splitting
  * If a customer checks out with products from multiple sellers, we must either:
    1. Split the cart into sub-orders at checkout, allowing the customer to select a delivery provider per seller.
    2. Default all items to their respective sellers' Self-Delivery configurations.

B. Fallback & Out-of-Service Zones
  * If a partner API is down or returns "not serviceable" for the target address:
    * The system should automatically fall back to "Seller Self-Delivery" as the primary option, ensuring checkouts are not blocked.

C. Cancellation & Partial Refunds
  * If an order is cancelled:
    * Check if a partner delivery was dispatched. If yes, issue a cancellation request via the adapter.
    * If the carrier charges a cancellation fee, deduct that from the customer's refund and record it in the platform logistics ledger.
